# 개발 기본 가이드

이 문서는 새 프로젝트를 시작하기 전에 개발자가 읽는 기준 문서다.
AI에게 직접 입력하는 프롬프트가 아니라, 사람이 판단할 때 참고한다.

AI에게 실제로 전달할 실행 명령은 `prompts/` 문서를 사용한다.

## 먼저 결정할 것

코드를 만들기 전에 아래 항목을 먼저 정한다.

- 무엇을 만들 것인가
- 누가 사용할 것인가
- v1에서 반드시 필요한 것은 무엇인가
- 이번에 하지 않을 것은 무엇인가
- 실패하면 어떤 문제가 생기는가
- 완료 기준은 무엇인가

이 항목이 불명확하면 코드를 먼저 만들지 않는다.

## 기본 원칙

### MUST

- 민감정보를 코드, 문서, 로그에 남기지 않는다.
- 입력과 출력의 형식, 단위, 실패 조건을 명확히 한다.
- 예외를 조용히 무시하지 않는다.
- 설정값을 코드 곳곳에 하드코딩하지 않는다.
- 외부 API, DB, 파일, 장비 연결부와 핵심 로직을 분리한다.

### SHOULD

- 작은 프로젝트는 작은 구조로 시작한다.
- 클래스는 상태와 책임을 묶을 이유가 있을 때만 만든다.
- 추상화는 실제 변경 지점이 보일 때 만든다.
- 핵심 로직은 테스트 가능한 형태로 둔다.
- 로그는 나중이 아니라 처음부터 최소한으로 둔다.

### AVOID

- 미래에 필요할 것 같다는 이유로 구조를 먼저 크게 만들기
- 단순 스크립트에 과한 객체지향 적용하기
- 보지 않은 파일을 기준으로 AI에게 수정하게 하기
- 테스트 없이 AI 결과를 믿기
- 전체 구조를 한 번에 갈아엎기

## 구조 판단 기준

처음부터 복잡한 구조를 만들지 않는다.

단일 파일로 충분한 경우:

- 일회성 자동화다.
- 입력과 출력이 단순하다.
- 외부 의존성이 거의 없다.
- 혼자 사용한다.

폴더 구조가 필요한 경우:

- 기능이 여러 개로 나뉜다.
- 테스트가 필요하다.
- 설정값이 많다.
- 외부 API, DB, 장비 연결이 있다.
- 여러 사람이 유지보수한다.

## AI 사용 기준

AI에게 바로 "만들어줘"라고 하지 않는다.

좋은 순서:

1. 목적과 범위를 정리한다.
2. 프로젝트 타입을 판단하게 한다.
3. 관련 파일 후보를 고르게 한다.
4. 필요한 파일만 제공한다.
5. 큰 변경이면 설계안을 먼저 받는다.
6. 최소 수정으로 구현하게 한다.
7. 검증 프롬프트로 다시 확인한다.

AI에게 반드시 지시할 것:

- 보지 않은 파일은 추측하지 말 것
- 부족한 파일은 먼저 요청할 것
- 전체 구조를 임의 변경하지 말 것
- 불필요한 라이브러리를 추가하지 말 것
- 수정 이유와 검증 방법을 함께 제시할 것

## 컨텍스트 제공 순서

AI에게 긴 코드베이스를 줄 때는 이 순서를 따른다.

1. 사용자 요청
2. 프로젝트 요약
3. 파일 구조
4. 직접 관련 파일
5. import된 내부 모듈
6. 테스트 파일
7. 설정 파일

전체 코드를 한 번에 넣는 것보다, 관련 파일을 좁혀 가는 방식이 낫다.

## 작업 전 체크리스트

- [ ] 목적이 한 문장으로 설명되는가?
- [ ] v1 범위가 정해졌는가?
- [ ] 하지 않을 일이 정해졌는가?
- [ ] 프로젝트 타입에 맞는 `profiles/` 문서를 골랐는가?
- [ ] 입력, 출력, 실패 조건이 정해졌는가?
- [ ] 설정값을 어디에 둘지 정했는가?
- [ ] 테스트 또는 수동 검증 방법이 있는가?
- [ ] 민감정보가 들어갈 가능성을 확인했는가?
- [ ] 지금 구조가 실제 필요보다 크지 않은가?
