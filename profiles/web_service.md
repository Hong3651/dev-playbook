# 프로필: 웹 서비스

API 서버, 대시보드, annotation UI, admin tool, 내부 웹 도구처럼 사용자 입력과 네트워크 요청을 다루는 작업에 적용한다.

## 선택 기준

이 프로필을 고르는 경우:

- 사용자가 브라우저, API, 대시보드, admin tool을 통해 접근한다.
- 인증, 권한, 입력 검증, 에러 응답이 필요하다.
- 서버 상태, DB, 파일 업로드, 외부 API와 연결된다.
- 운영 중인 사용자의 요청을 안정적으로 처리해야 한다.

이 프로필만으로 부족한 경우:

- 웹 화면이 실제 장비 명령이나 inference 실행을 트리거하면 `robot_infra.md`를 함께 적용한다.
- 단순 내부 파일 변환 도구라면 `automation.md`로 작게 시작한다.
- 공개 서비스라면 배포, 모니터링, 개인정보, 법적 요구사항을 별도로 검토한다.

애매한 경우에는 외부 입력과 권한이 있는지로 판단한다.
사용자가 값을 넣고 시스템 상태가 바뀐다면 web_service 기준을 적용한다.

## 적용 범위

- API 서버
- 내부 대시보드
- annotation 또는 labeling UI
- admin tool
- 실험 결과 조회 도구
- 모델/장비 상태 모니터링 웹 도구

## 기본 기준

사용자 입력은 신뢰하지 않는다.
프론트엔드에서 막았더라도 백엔드에서 다시 검증한다.

반드시 고려할 것:

- 인증
- 인가
- 입력값 검증
- API 계약
- 에러 응답 구조
- 환경변수 관리
- 로그의 민감정보 제거

## 프론트엔드와 백엔드 책임

프론트엔드:

- 사용자가 이해하기 쉬운 입력 흐름 제공
- 기본적인 입력 형식 안내
- 로딩, 실패, 빈 상태 표시
- API 응답에 따른 화면 처리

백엔드:

- 권한 검증
- 입력값 최종 검증
- 데이터 저장과 조회
- 외부 API/DB 연동
- 일관된 에러 응답 제공

## API 기준

- 요청/응답 스키마를 문서화한다.
- 실패 응답은 일관된 형식으로 반환한다.
- 내부 예외 상세를 사용자에게 그대로 노출하지 않는다.
- breaking change가 있으면 명확히 기록한다.

예시:

```json
{
  "error": {
    "code": "INVALID_INPUT",
    "message": "입력값이 올바르지 않습니다."
  }
}
```

## 보안 기준

- 토큰, 세션, 쿠키 정책을 명확히 한다.
- 권한 체크를 화면 표시 여부에만 의존하지 않는다.
- SQL injection, XSS, CSRF, 파일 업로드 위험을 검토한다.
- 로그에 개인정보, 토큰, 내부 URL을 남기지 않는다.
- 환경변수 누락 시 안전하게 실패한다.
